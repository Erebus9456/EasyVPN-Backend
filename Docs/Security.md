## Security Philosophy

EasyVPN is designed around a simple principle:

> **Never expose what does not need to be exposed.**

The system assumes:

* VPS nodes are semi-trusted infrastructure
* The backend is orchestration-only
* WireGuard is the primary security boundary

---

## Threat Model

EasyVPN protects against:

* Unauthorized VPN access
* Rogue peer injection
* API abuse
* Node impersonation
* Registry tampering

It does NOT attempt to solve:

* Physical server compromise
* Kernel-level attacks on VPS
* Malicious cloud provider interference

---

## System Security Boundaries

| Layer       | Responsibility     | Trust Level            |
| ----------- | ------------------ | ---------------------- |
| WireGuard   | Traffic encryption | Trusted                |
| VPS Agent   | Peer management    | Semi-trusted           |
| Backend API | Orchestration      | Trusted                |
| Supabase    | Registry only      | Untrusted for security |

---

## API Security

### Token Authentication

All sensitive endpoints require:

```http id="sec01"
X-API-TOKEN: <secret>
```

* Token is defined in `.env`
* Stored only on the VPS
* Never exposed to clients

---

### Protection Scope

The API token protects:

* `/add-peer`
* `/replace-peer`
* Peer management actions
* WireGuard modifications

---

## Network Security

### Exposed Ports

| Port  | Protocol | Exposure                    |
| ----- | -------- | --------------------------- |
| 5000  | TCP      | Internal control plane only |
| 51820 | UDP      | Public VPN tunnel           |

---

### Critical Rule

> Only WireGuard (51820/UDP) should be exposed publicly.

Port 5000 must be restricted via:

* Cloud firewall (AWS / Azure / GCP / DO)
* Security groups / NSGs

## WireGuard Security Model

WireGuard provides:

* Strong cryptographic identity per peer
* Noise-based handshake protocol
* Kernel-level packet encryption
* No plaintext traffic exposure

Each peer has:

* Unique keypair
* Dedicated IP (10.0.0.x/32)
* Isolated routing entry

---

## Key Management

### Server Keys

* Generated during provisioning
* Stored in protected filesystem (`root-only`)
* Never transmitted outside VPS

---

### Client Keys

* Generated by backend control plane
* Only public keys are sent to VPS agent
* Private keys remain on user devices

---

## Data Protection Model

### At Rest

* Peer state stored in `peers.json`
* Server identity stored in `state.json`
* Files protected via root permissions

---

### In Transit

* All VPN traffic encrypted via WireGuard
* API communication secured via token header
* No plaintext credentials over network

---

## Attack Surface Reduction

### 1. Minimal Open Ports

Only two ports exist:

* 5000 (internal API)
* 51820 (VPN tunnel)

---

### 2. No Public Admin Panel on VPS

All management happens via backend control plane.

---

### 3. Idempotent Operations

Repeated provisioning:

* Does not duplicate peers
* Does not regenerate keys unnecessarily
* Does not corrupt configuration state

---

### 4. Systemd Hardening

Agent service is configured with:

* Auto-restart on crash
* Delayed restart backoff
* Persistent execution after reboot

---

## Isolation Strategy

### Control Plane Isolation

If backend is compromised:

* Existing VPN tunnels remain unaffected
* WireGuard continues operating independently

---

### Node Isolation

If one VPS is compromised:

* Only that node is affected
* Other nodes remain unaffected via Supabase registry

---

## Secure Deployment Checklist

Before production:

* [ ] TCP 5000 restricted via firewall
* [ ] UDP 51820 open globally
* [ ] `.env` never committed to git
* [ ] Root-only file permissions enforced
* [ ] systemd service enabled
* [ ] Supabase keys secured

---

## Common Security Mistakes

### ❌ Exposing Port 5000 publicly

This allows unauthorized peer injection.

---

### ❌ Editing WireGuard manually

Breaks consistency with agent state.

---

### ❌ Sharing `.env`

Leaks API token and registry access.

---

## Recommended Hardening (Optional)

For advanced deployments:

* Restrict port 5000 to backend IP only
* Use VPC/private networking between backend and VPS
* Enable fail2ban (optional layer)
* Monitor `wg show` for anomalies

---

## Security Summary

EasyVPN security relies on:

* WireGuard cryptography
* Strict API token protection
* Minimal network exposure
* Isolated node architecture
* Stateless control plane design
